# Contributing Guidelines

Thank you for your contribution to the project name. Your help is very important in advancing this project.

## Community Code of Conduct
Our project is committed to ensuring a safe and welcoming environment for all participants to contribute.
All contributors must comply with the Code of Conduct outlined in **[CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md)**. Violations should be reported according to the procedures specified in that document.

## How to contribute

There are several ways you can contribute to the project.

1. Report a bug: Let me know if you find an error.
2. Feature Request: Please suggest a new idea.
3. CODE CONTRIBUTION: Fix the bug or implement a new feature.
4. Document improvement: Please improve README or other documents.


## 1. Report bugs and request functions

Before creating a new issue, please search the existing issue list to see if there are any duplicates.

### Report a bug

Please include the following information when creating an issue.

* Description: Briefly explain what a bug is.
* Reproduction Steps: List in detail the series of steps in which the bug occurs.
* Expected Results: What were the expected outcomes at that stage?
* Actual Results: What are the actual results that happened?
* Environment: Specifies your environment (OS, browser, software version, etc.).

### Feature Request

Please include the following when requesting new features.

* Summary: Summarize features in one sentence.
* Necessity: Explain why this feature is needed and what issues it solves.
* Implementation Proposition: Briefly describe the proposed technical implementation, if possible.


## 2. Code Contribution Workflow

Please follow the steps below to contribute to the code.

### 1. Storage Fork and Clones

1. Fork the project repository to your account on GitHub.
2. Clones the forked repository to the local machine.

### 2. Create Branch

* All work should be done in a new branch, not in the 'main' branch.
* Branch Naming Conventions:
* Fixing bugs: 'fix/issue number-simple description' (e.g. 'fix/15-login-error')
* Added features: 'feature/simple explanation' (e.g. 'feature/add-dark-mode')

### 3. Commit Message Rules

* The commit message must be clear and concise.
* The first line shall be made up of no more than 50 characters, using the present tense (e.g., 'Add login feature').
* Leave the second line blank, and you can create details from the third line.
* If you fix a bug, you can specify that the issue should be closed automatically, including the issue number (e.g. 'Fixes #issueNumber')

### 4. Submit a Pull Request (PR)

1. When you are done, push to the forked repository.
2. Create a Pull Request with the 'main' branch of theproject repository.
3. PR The PR body must specify the task and the issue number resolved.

The project management team will review the submitted PR and provide feedback.

# 기여 가이드라인

프로젝트명에 기여해 주셔서 감사합니다. 여러분의 도움은 이 프로젝트를 발전시키는 데 매우 중요합니다.

## 기여 방법

프로젝트에 기여할 수 있는 몇 가지 방법이 있습니다.

1.  버그 신고: 오류를 발견하면 알려주세요.
2.  기능 요청: 새로운 아이디어를 제안해 주세요.
3.  코드 기여: 버그를 수정하거나 새로운 기능을 구현해 주세요.
4.  문서 개선: README나 기타 문서를 개선해 주세요.

## 1. 버그 신고 및 기능 요청

새로운 이슈를 생성하기 전에 기존 이슈 목록을 검색하여 중복되는 내용이 없는지 확인해 주세요.

### 버그 신고

이슈를 생성할 때 다음 정보를 포함해 주세요.

* 설명: 버그가 무엇인지 간결하게 설명합니다.
* 재현 단계: 버그가 발생하는 일련의 단계를 상세히 나열합니다.
* 기대 결과: 해당 단계에서 예상되는 결과는 무엇이었습니까?
* 실제 결과: 실제로 발생한 결과는 무엇입니까?
* 환경: 사용 환경(OS, 사용 브라우저, 소프트웨어 버전 등)을 명시합니다.

### 기능 요청

새로운 기능 요청 시에는 다음을 포함해 주세요.

* 요약: 한 문장으로 기능을 요약합니다.
* 필요성: 이 기능이 왜 필요한지, 어떤 문제를 해결하는지 설명합니다.
* 구현 제안: 가능하다면 제안하는 기술적 구현 방식을 간략히 설명합니다.


## 2. 코드 기여 워크플로우

코드 기여를 하려면 아래 단계를 따라주세요.

### 1. 저장소 포크(Fork) 및 클론(Clone)

1.  GitHub에서 프로젝트 저장소를 자신의 계정으로 포크(Fork)합니다.
2.  포크한 저장소를 로컬 머신에 클론합니다.

### 2. 브랜치 생성

* 모든 작업은 `main` 브랜치가 아닌 새로운 브랜치에서 진행해야 합니다.
* 브랜치 명명 규칙:
    * 버그 수정: `fix/이슈번호-간단설명` (예: `fix/15-login-error`)
    * 기능 추가: `feature/간단설명` (예: `feature/add-dark-mode`)

### 3. 커밋 메시지 규칙

* 커밋 메시지는 명확하고 간결해야 합니다.
* 첫 줄은 50자 이내로 작성하며, 현재 시제(예: `Add login feature`)를 사용합니다.
* 두 번째 줄은 비워두고, 세 번째 줄부터 상세 내용을 작성할 수 있습니다.
* 버그를 수정한 경우, 이슈 번호를 포함하여 자동으로 이슈를 닫도록 명시할 수 있습니다. (예: `Fixes #이슈번호`)

### 4. 풀 리퀘스트(Pull Request, PR) 제출

1.  작업을 마쳤으면 포크한 저장소에 푸시(Push)합니다.
2.  프로젝트 저장소의 `main` 브랜치로 풀 리퀘스트(Pull Request)를 생성합니다.
3.  PR 본문에는 작업 내용과 **해결한 이슈 번호**를 반드시 명시해야 합니다.

프로젝트 관리팀은 제출된 PR을 검토하고 피드백을 제공할 것입니다.
